# BMO Voice - Technology Decisions

Every component choice below is justified against the goals: **minimize latency, maximize accuracy, zero cost, fully open source**.

Items marked with ◆ are specific model/service choices — review these to decide if you'd prefer an alternative.

## 1. Wake Word Detection: ◆ OpenWakeWord

| Considered | Open Source | Custom Word Support | Why not chosen |
|------------|-------------|---------------------|----------------|
| **◆ OpenWakeWord** | Yes (Apache 2.0) | Yes, trainable | **CHOSEN** |
| Porcupine (Picovoice) | No (proprietary) | Yes | Proprietary, free tier limited |
| Mycroft Precise | Yes | Yes | Less actively maintained |

**Why OpenWakeWord:**
- Fully open source, no API keys, no telemetry
- Custom wake word training from synthetic speech (no need to record thousands of samples)
- Train "Hey Beemo" model in < 1 hour using provided Colab notebook
- Lightweight: runs on CPU with ONNX runtime (~2% CPU, ~50MB RAM)
- Python native, easy to integrate
- Runs continuously in background with minimal resource usage

**Why not just use Whisper to detect "Hey Beemo"?**
Running Whisper continuously to transcribe all audio and check for "Hey Beemo" would consume ~80% CPU constantly. OpenWakeWord is a tiny binary classifier (~50MB) that answers "is this 'Hey Beemo'?" with ~2% CPU. It's the same approach used by Alexa, Google Home, and Siri — a dedicated lightweight model for wake word detection, separate from the full speech recognition system.

**Wake word: "Hey Beemo"**
- Train a custom model using OpenWakeWord's training pipeline
- **Speaker-independent**: Detects the phrase from *any* speaker, not tied to a specific voice
- All training data is 100% synthetic (generated by TTS models with diverse voices/accents)
- Generate ~1000+ synthetic "Hey Beemo" clips + adversarial negatives ("hey baby", "hey maybe", etc.)
- Train small classifier on top of OpenWakeWord's frozen feature extractor
- Training takes < 1 hour on CPU, produces a lightweight `.onnx` model

**If OpenWakeWord is removed/breaks:**
- Fallback to keyword spotting using whisper.cpp (listen for "hey beemo" in continuous short transcripts)
- Higher CPU usage but no additional dependency

## 3. Speech-to-Text (STT): ◆ whisper.cpp with MLX backend

| Considered | Open Source | Local | Latency (M1 Pro) | Why not chosen |
|------------|-------------|-------|-------------------|----------------|
| **◆ whisper.cpp (MLX)** | Yes (MIT) | Yes | ~500-800ms | **CHOSEN** |
| faster-whisper | Yes | Yes | ~600-1000ms | CTranslate2 less optimized for Apple Silicon than MLX |
| Deepgram Nova-3 | No | No | ~200ms | Paid service |
| WhisperKit (Swift) | Yes | Yes | ~460ms | Swift-only, harder to integrate with Python server |

**Why whisper.cpp with MLX:**
- Fully open source, runs 100% locally
- MLX backend is specifically optimized for Apple Silicon (M1/M2/M3/M4)
- Faster than real-time on M1 Pro with base/small model
- Core ML acceleration available for additional speed
- No API keys, no network, no cost

### ◆ Whisper Model Comparison

#### Decision Point: English-only `.en` vs Multilingual `turbo`

The English-only `.en` models are smaller and traditionally faster for English. However, the newer `turbo` model (multilingual) achieves near-best accuracy at competitive speed. **Benchmark both on your hardware to decide.**

| Model | Params | RAM | English WER | Speed (M1 Pro est.) | Notes |
|-------|--------|-----|-------------|----------------------|-------|
| **small.en** | 244M | ~852 MB | ~3.4% | ~0.15x RT | **Best English-only balance** |
| **turbo** | 809M | ~2.3 GB | ~2.5% | ~0.2x RT | **Best multilingual, near-large accuracy** |
| tiny.en | 39M | ~273 MB | ~7.6% | ~0.05x RT | Too inaccurate for voice assistant |
| base.en | 74M | ~388 MB | ~5.0% | ~0.1x RT | Acceptable accuracy, fastest |
| medium.en | 769M | ~2.1 GB | ~2.9% | ~0.4x RT | High accuracy, noticeable wait |
| large-v3 | 1,550M | ~3.9 GB | ~2.4% | ~1.0x RT | Near real-time limit |

*RT = Real-time. 0.2x RT means a 5-second utterance takes ~1 second to transcribe.*

**What the numbers mean for a typical 3-5 second voice query:**

| Model | Time to transcribe 5s audio | Error rate |
|-------|-----------------------------|--------------------|
| **small.en** | **~0.75-1.0s** | **~1 in 30 words wrong** |
| **turbo** | **~1.0-1.5s** | **~1 in 40 words wrong** |
| medium.en | ~1.5-2.0s | ~1 in 34 words wrong |

### ◆ Recommended: whisper-small.en (default), with turbo as alternative

- **small.en**: 3.4% WER, ~0.75-1.0s latency, 852 MB RAM — best speed/accuracy balance
- **turbo**: 2.5% WER, ~1.0-1.5s latency, 2.3 GB RAM — better accuracy, +0.3-0.5s latency
- The model is swappable via config — no code changes needed
- **Action item**: Benchmark both on M1 Pro and compare. If turbo is noticeably better in real-world use, switch.

**If whisper.cpp is removed/breaks:**
- faster-whisper (CTranslate2) as direct replacement

## 4. Large Language Model (LLM): ◆ Ollama + Qwen3 8B

| Considered | Open Source | Local | Speed (M1 Pro) | Tool Use | Why not chosen |
|------------|-------------|-------|-----------------|----------|----------------|
| **◆ Ollama + Qwen3 8B** | Yes | Yes | ~30-50 tok/s | Yes | **CHOSEN** |
| Ollama + Llama 3.3 8B | Yes | Yes | ~30-50 tok/s | Yes | Slightly lower quality than Qwen3 for instruction following |
| Ollama + Mistral Small 3 7B | Yes | Yes | Highest tok/s | Limited | Best speed but weaker tool use |
| Claude Sonnet | No | No | Fast | Excellent | Paid, not open source |

**Why Ollama + Qwen3 8B:**
- Fully open source, runs 100% locally via Ollama
- ◆ **Qwen3 8B (Q4_K_M quantization)** — ~5GB RAM, fits comfortably on M1 Pro 32GB
- Best-in-class instruction following and creative conversation for 8B class
- Supports tool/function calling (needed for intent routing)
- /think mode available for complex queries (chain-of-thought)
- ~30-50 tokens/second on M1 Pro — fast enough for streaming to TTS
- Ollama handles model management, quantization, and inference

**◆ Alternative model options (same Ollama infrastructure):**
- **Qwen3.5 9B** — newer, potentially better quality, 6.6GB
- **Llama 3.3 8B** — Meta's model, strong all-around
- **Mistral Small 3 7B** — fastest tok/s if latency is top priority

**If Ollama is removed/breaks:**
- llama.cpp directly (Ollama is a wrapper around it)
- LM Studio as alternative local inference server

- Sub-300ms processing, runs at 36x real-time on GPU (CPU still fast)

**If Kokoro is removed/breaks:**

## 6. RAG - Embedding Model: ◆ BGE-small-en-v1.5 (local)

| Considered | Open Source | Local | Latency (per query) | Dimensions | Why not chosen |
|------------|-------------|-------|---------------------|------------|----------------|
| **◆ BGE-small-en-v1.5** | Yes (MIT) | Yes | ~10-20ms | 384 | **CHOSEN** |
| all-MiniLM-L6-v2 | Yes | Yes | ~10ms | 384 | Slightly lower retrieval quality |
| BGE-M3 | Yes | Yes | ~50ms | 1024 | Overkill; multilingual not needed |
| OpenAI text-embedding-3-small | No | No | ~20ms (API) | 1536 | Paid API, not open source |

**Why BGE-small-en-v1.5:**
- Fully open source (MIT), runs locally via sentence-transformers
- Optimized for English (our only language)
- 384 dimensions — fast similarity search, low storage
- ~10-20ms per embedding — fast enough for real-time query embedding
- Runs on CPU efficiently (no GPU required)

**If BGE is removed/breaks:**
- all-MiniLM-L6-v2 as drop-in replacement (same dimension, similar API)
- Ollama also supports embedding models (nomic-embed-text)

## 7. RAG - Vector Database: ◆ ChromaDB (local)

| Considered | Open Source | Local | Latency | Why not chosen |
|------------|-------------|-------|---------|----------------|
| **◆ ChromaDB** | Yes (Apache 2.0) | Yes | < 5ms | **CHOSEN** |
| FAISS | Yes | Yes | < 5ms | Lower-level, no metadata filtering |
| LanceDB | Yes | Yes | < 5ms | Newer, less ecosystem |
| Qdrant | Yes | Yes (self-host) | < 10ms | Heavier, separate process |

**Why ChromaDB:**
- Open source (Apache 2.0), runs in-process
- Zero network latency, zero cost
- Built-in metadata filtering (filter by folder, file type)
- Persistent storage to disk
- HNSW indexing for fast approximate nearest neighbor

## 9. Web Search: ◆ Brave Search API (free tier) with ◆ SearXNG fallback

| Considered | Open Source | Free | Query Limits | Why not chosen |
|------------|-------------|------|--------------|----------------|
| **◆ Brave Search (free tier)** | No (free API) | 2000/mo free | 2000 queries/mo | **CHOSEN (primary)** |
| **◆ SearXNG (self-hosted)** | Yes (AGPL) | Unlimited | None | **CHOSEN (fallback)** |
| DuckDuckGo (unofficial) | No | Unofficial | Unreliable | Could break anytime |
| Tavily | No | Limited free | 1000/mo | Less generous free tier |

**Why Brave Search (primary):**
- 2000 free queries/month (no credit card required)
- Highest benchmark score for LLM grounding
- Lowest latency among search APIs (669ms)
- Independent index, privacy-first

**Why SearXNG (fallback):**
- Fully open source, self-hosted, zero cost, unlimited queries
- Aggregates from Google, Bing, DuckDuckGo — broad coverage
- Runs via Docker on same machine
- **If Brave free tier is removed**: SearXNG becomes primary with no loss of functionality

**Free tier risk assessment:**
- Brave's 2000/mo free tier has been stable since launch
- Even at heavy use (20 searches/day), that's 600/month — well within limits

## The Latency Budget (Fully Local)

```
"Hey Beemo" detected              0ms
├─ Wake word processing            ~150ms
├─ User speaks (VAD listening)     variable
├─ VAD detects end of speech       ~100ms
├─ whisper.cpp STT                 ~500-800ms  (depends on utterance length)
├─ Intent routing                  ~10ms
├─ RAG retrieval (if needed)       ~30ms   (local ChromaDB + BGE)
├─ Web search (if needed)          ~700ms  (Brave API, parallel with RAG)
├─ Ollama LLM TTFT                 ~200-300ms
├─ Kokoro TTS TTFB                 ~200-300ms
└─ Audio buffer                    ~50ms
                                   ────────
Total (general query):             ~1200-1600ms
Total (RAG query):                 ~1250-1650ms  (RAG is fast, local)
Total (web search query):          ~1600-2000ms  (web adds network latency)
```
