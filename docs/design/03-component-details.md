# BMO Voice - Component Details

Items marked with ◆ are specific model/service choices — see [02-technology-decisions.md](02-technology-decisions.md) for alternatives.

---

## Component 1: Wake Word Detection

### Responsibilities
- Continuously listen for "Hey Beemo" while idle
- On detection, activate the voice pipeline (VAD → STT → LLM → TTS)
- Low CPU usage during idle listening
- Reject false positives (TV, music, similar-sounding words)

### Implementation Details
- **◆ OpenWakeWord** running in a dedicated thread
- Process audio in 80ms frames (OpenWakeWord's default)
- Detection threshold: tunable (start at 0.5, adjust based on false positive rate)
- On detection: emit event to session manager, begin recording audio for STT
- During TTS playback: still listen for "Hey Beemo" as an interruption signal
- **Debounce**: Ignore duplicate wake word detections within 500ms

### Custom Model Training
- **Speaker-independent**: Detects "Hey Beemo" from any speaker, not tied to a specific voice
- All training data is 100% synthetic — generated by TTS models with diverse voices and accents
- Generate ~1000+ synthetic "Hey Beemo" clips using multiple TTS voices
- Generate adversarial negative samples ("hey baby", "hey maybe", "hey people", etc.)
- Add room impulse responses and background noise augmentation for robustness
- Train small classifier on OpenWakeWord's frozen feature extractor
- Training takes < 1 hour on CPU, produces a lightweight `.onnx` model
- Store trained model at `~/.bmo-voice/models/hey_beemo.onnx`

### Edge Cases
- **TV/music playing**: Threshold tuning + adversarial training reduces false positives
- **Multiple rapid detections**: 500ms debounce prevents duplicate triggers
- **Wake word during TTS playback**: Must still detect — use echo reference or separate mic channel

## Component 2: Voice Activity Detection (VAD)

### Responsibilities
- After wake word or in conversation mode, detect when the user starts and stops speaking
- Provide end-of-turn signal to trigger STT processing
- Filter background noise during active listening

### Implementation Details
- **◆ Silero VAD** running in the Python server process
- Process audio in 30ms frames
- Speech start: 3 consecutive frames above threshold → begin buffering audio
- Speech end: 500ms of silence (configurable) → trigger STT
- Adaptive threshold based on ambient noise (calibrate at session start)

### Edge Cases
- **Filler words** ("um", "uh"): Don't trigger end-of-turn on pauses < 300ms
- **Interruption**: If VAD detects speech while TTS is playing, signal interrupt
- **Back-to-back utterances**: Allow 200ms grace period before returning to idle
- **No speech after wake word**: 5-second timeout → return to idle, optionally play subtle tone
- **Very long utterance (>30s)**: Segment at 30-second boundaries for Whisper processing, concatenate transcripts
- **Minimum speech duration**: Require 500ms of continuous speech before processing (prevents noise triggers in conversation mode)

## Component 3: Speech-to-Text (STT)

### Responsibilities
- Convert buffered audio to text after VAD signals end-of-turn
- Fast, accurate transcription of English speech

### Implementation Details
- **◆ whisper.cpp with MLX backend** for Apple Silicon optimization
- **◆ Model: whisper-small.en** (English-only, ~460MB) — turbo model as alternative (see 02-technology-decisions.md)
- Process complete utterance buffer (not streaming — wake word + VAD already segment the audio)
- Output: full transcript text

### Configuration
```
model: whisper-small.en       # or whisper-turbo (benchmark to decide)
backend: mlx (Apple Silicon) or cpu (fallback)
language: en
beam_size: 5
```

### Optimization
- Keep model loaded in memory (don't reload per utterance)
- MLX uses Apple's unified memory — no CPU↔GPU transfer overhead
- English-only model is faster and more accurate than multilingual for English
- Pre-allocate audio buffer to avoid allocation during recording

### Edge Cases
- **Empty/mumbled input**: If transcript is empty or very short (<3 characters), respond with "Sorry, I didn't catch that"
- **Background noise transcribed as words**: STT confidence filtering — discard low-confidence results
- **Long utterances (>30s)**: Process in segments, concatenate

## Component 4: Intent Router

### Responsibilities
- Analyze transcript to determine which tools/sources are needed
- Route to the optimal execution path
- Handle follow-up queries by checking conversation context

### Implementation Details
- **Rule-based first pass** (near-zero latency):
  - Explicit triggers: "look up...", "google...", "search for..." → web search
  - Explicit triggers: "in my documents...", "from my files...", "in [folder name]..." → RAG
  - Follow-up within conversation context to previous RAG/search → same source
  - Everything else → general (direct LLM)

- **LLM classification (when ambiguous)**:
  - Use ◆ **Ollama (same Qwen3 8B)** with a short classification prompt
  - System prompt: classify as `general`, `rag`, `web_search`, or `hybrid`
  - Only used when rule-based routing is uncertain

### Execution Strategies
| Intent | Execution |
|--------|-----------|
| `general` | Ollama LLM direct (streaming) |
| `rag` | Embed query → ChromaDB search → Ollama with context |
| `web_search` | Brave/SearXNG search → Ollama with results |
| `hybrid` | RAG + web search **in parallel** → Ollama with both contexts |

### Follow-Up Handling
When in conversation mode and a follow-up is detected:
1. Check if the query references the previous turn (pronouns, continuation phrases)
2. If previous turn used RAG/web search, default to same source
3. Include previous Q&A pair in the LLM context for coherent follow-ups
4. Allow source switching: "Actually, search the web for that instead"

## Component 5: RAG Engine

### Responsibilities
- Index user-specified folders into vector database (text content only)
- Retrieve relevant chunks for user queries
- Keep index up-to-date as files change

### Sub-components

#### 5a. Document Loader
- **Supported formats**: .txt, .md, .pdf, .docx, .py, .js, .ts, .json, .yaml, .csv, .html
- **Text extraction only**: Only text content is extracted from all formats. Images, diagrams, and non-text elements in PDFs/DOCX are skipped. BMO responds based on whatever text is available — no special messaging about images.
- **Library**: `unstructured` for PDF/DOCX text extraction, raw read for text/code files
- **Metadata per document**: file path, file type, folder name, last modified, title

#### 5b. Chunker
- **Strategy**: Recursive character splitting at 512 tokens
- **Overlap**: 50 tokens (~10%)
- **Separators** (priority order): `\n\n`, `\n`, `. `, ` `, ``
- **Code files**: Split at function/class boundaries when possible (tree-sitter for AST-aware splitting)
- **Metadata per chunk**: source file, chunk index, character offset, folder name

#### 5c. Embedding & Storage
- **◆ Embedding model: BGE-small-en-v1.5** via sentence-transformers (local, 384 dimensions)
- **◆ Vector DB: ChromaDB** with persistent storage at `~/.bmo-voice/chroma/`
- **Collections**: One collection per registered folder
- **Batch embedding**: Process chunks in batches of 32 during indexing
- **Parallel indexing**: Multiple files can be chunked and embedded concurrently

#### 5d. Retrieval
- Embed query with same BGE model → ChromaDB similarity search
- Top-k: 5 chunks by default (configurable)
- **Metadata filtering**: Restrict to specific folders if user mentions one by name
- Total context: ~2500 tokens from top-5 chunks

#### 5e. File Watcher
- **Library**: `watchdog` (Python, open source)
- Watches all registered folders for file changes
- On change: re-chunk and re-embed only the affected file
- On delete: remove chunks for that file from vector DB
- Debounce: 2-second delay to batch rapid saves

### Edge Cases
- **RAG folder deleted while indexed**: File watcher detects deletion, removes chunks from ChromaDB, logs warning
- **Very large files**: Chunk normally but cap at 1000 chunks per file to prevent index bloat
- **Binary files in watched folders**: Skip non-supported file types silently
- **Conflicting info across folders**: Return chunks from all relevant folders, let LLM synthesize

## Component 6: Web Search

### Responsibilities
- Search the web for current/real-time information
- Format results as context for the LLM

### Implementation Details
- **Primary: ◆ Brave Search API** (free tier, 2000 queries/month)
  - Query → top 5 results with title, URL, snippet
  - Use snippets only (fastest, no page fetching needed)
  - If snippets insufficient, fetch top 1-2 pages via `httpx` + `readability-lxml`

- **Fallback: ◆ SearXNG** (self-hosted via Docker)
  - Same query interface, returns aggregated results from multiple engines
  - No query limits, no API key

### Context Assembly
- Format: `[Source: title](url) — snippet text`
- Total web context: ~1500 tokens from top 3-5 results
- Include source URLs so LLM can cite them verbally

### Edge Cases
- **Network failure**: 3-second timeout, proceed without web context, inform user: "I couldn't reach the web right now"
- **Brave rate limit hit**: Automatic fallback to SearXNG
- **Empty results**: Tell user "I couldn't find anything about that online"

## Component 7: LLM Response Generation

### Responsibilities
- Synthesize context (RAG, web search, conversation history, user preferences) into a spoken response
- Stream tokens for TTS processing

### Implementation Details
- **◆ Ollama + Qwen3 8B** via local HTTP API (streaming)
- Ollama runs as a background service on the same machine
- Streaming via Ollama's `/api/chat` endpoint with `stream: true`

### System Prompt (Voice-Optimized)
```
You are BMO, a helpful voice assistant. You speak out loud to the user.

Rules:
- Be concise. 1-3 sentences unless asked for detail.
- Use natural spoken English, not written language.
- No markdown, bullet points, or formatting — this will be spoken aloud.
- Don't preface answers with "according to your documents" — just answer naturally.
- If unsure, say so briefly: "I'm not sure about that."
- Numbers: say "about two hundred" not "approximately 200".
- When citing sources, say "based on [filename]" or "I found online that..."
- For multi-part questions, structure as: "First... Second... And finally..."
- If a query needs RAG or web search, acknowledge before the delay: "Let me check on that."
- If asked to do something you can't do, say "I can't do that yet" — don't make up capabilities.
- Speak dates naturally: "March twenty-first" not "2026-03-21".
- For lists, pace naturally: "There are three things. First... second... and third..."

User context:
- Name: {user_name}
- Location: {user_location}
- Temperature units: {temp_units}
- Distance units: {distance_units}
- Timezone: {timezone}
```

### Conversation Memory (5-minute TTL)
- Store conversation turns in an in-memory list with timestamps
- Before each LLM call, prune messages older than 5 minutes
- No persistence to disk — memory resets on restart or after 5 minutes of silence
- Typically holds last 5-15 turns depending on conversation pace

### Context Window Management
- Conversation history (within 5-min window): ~2000 tokens
- RAG context: top 5 chunks, ~2500 tokens
- Web search context: 3-5 results, ~1500 tokens
- System prompt + user context: ~300 tokens
- Total per request: ~4000-6300 tokens — well within 8B model context window (typically 8K-32K)

### Streaming Pipeline (LLM → TTS pipeline parallelism)
```
Ollama token stream → sentence buffer → Kokoro TTS → client playback
```
- Buffer tokens until sentence boundary (. ? ! \n)
- Send complete sentences to TTS for better prosody
- If buffer exceeds 100 tokens without punctuation, flush anyway
- **Pipeline parallel**: While TTS synthesizes sentence N, LLM continues generating sentence N+1

### Edge Cases
- **Ollama crashes mid-response**: Detect connection drop, say "Sorry, I had a problem. Could you ask again?"
- **Very long LLM response**: Cap at ~500 tokens for voice (configurable), add "Would you like me to continue?"
- **Empty context from RAG/web**: LLM responds from general knowledge, mentions it couldn't find specific info

## Component 8: Text-to-Speech (TTS)

### Responsibilities
- Convert LLM text to natural-sounding speech locally
- Stream audio back to client

### Implementation Details
- **◆ Kokoro TTS (82M)** running locally
- Input: sentence strings from LLM stream
- Output: PCM audio chunks (16kHz 16-bit)
- Keep model loaded in memory between utterances

### Configuration
```
model: kokoro-82m
voice: ◆ default English neutral (specific voice TBD after testing)
sample_rate: 16000
format: pcm_s16le
```

### Streaming Flow
1. Receive sentence from LLM buffer
2. Run Kokoro inference → audio array
3. Convert to PCM bytes
4. Send to client via WebSocket
5. Client begins playback immediately on first chunk

### Edge Cases
- **TTS fails on a sentence**: Skip that sentence, continue with next
- **Special characters or URLs in LLM output**: Strip or normalize before TTS
- **Very short sentences** ("Yes."): Batch with next sentence for better prosody

## Component 9: Conversation Mode & Session Management

### Responsibilities
- Manage WebSocket lifecycle with client
- Track conversation mode (wake-word-only vs active listening)
- Track 5-minute conversation history window
- Handle wake word → VAD → STT → LLM → TTS pipeline coordination

### Conversation Mode State Machine
```
┌─────────────────┐
│  IDLE           │  ← Wake-word-only mode (~2% CPU)
│  (listening for │
│   wake word)    │
└────────┬────────┘
         │ "Hey Beemo" detected
         ▼
┌─────────────────┐
│  LISTENING      │  ← VAD active, recording audio
│  (waiting for   │
│   user speech)  │
└────────┬────────┘
         │ VAD: end of speech (or 5s timeout → back to IDLE)
         ▼
┌─────────────────┐
│  PROCESSING     │  ← STT → Router → Context → LLM
│  (thinking)     │
└────────┬────────┘
         │ LLM tokens ready
         ▼
┌─────────────────┐
│  SPEAKING       │  ← TTS → audio playback
│  (responding)   │
└────────┬────────┘
         │ TTS complete
         ▼
┌─────────────────┐
│  CONVERSATION   │  ← Listening WITHOUT wake word for 30s
│  MODE           │     VAD active with min 500ms speech filter
│  (follow-up     │
│   window)       │
└────────┬────────┘
         │                              │
         │ Speech detected              │ 30s silence
         │ → go to PROCESSING           │ → go to IDLE
         ▼                              ▼
```

### State
```
Session:
  id: str
  websocket: WebSocket
  state: SessionState          # IDLE, LISTENING, PROCESSING, SPEAKING, CONVERSATION
  conversation_history: list[Message]  # pruned to 5-min TTL
  active_folders: list[str]            # folders registered for RAG
  current_task: asyncio.Task | None    # current pipeline run
  conversation_timer: Timer | None     # 30s conversation mode timeout
  user_preferences: UserPreferences    # name, location, units, timezone
```

### Interruption Handling
1. "Hey Beemo" detected during SPEAKING state
2. Cancel current LLM generation task
3. Stop TTS processing
4. Send stop signal to client audio player
5. Begin processing new utterance immediately (go to LISTENING)

### Conversation Mode Rules
- After TTS completes, enter CONVERSATION state for 30 seconds (configurable)
- In CONVERSATION state, VAD listens for speech without wake word
- Minimum 500ms of continuous speech required to trigger processing (prevents background noise false triggers)
- Each follow-up resets the 30s timer
- "Hey Beemo" always works in any state (including CONVERSATION)
- After 30s of silence in CONVERSATION → return to IDLE
- After 5 minutes of no interaction → conversation history cleared

### Edge Cases
- **Background noise in conversation mode**: 500ms minimum speech duration filters most noise
- **User talking to someone else in conversation mode**: Short window (30s) limits exposure; worst case is one misprocessed turn
- **User walks away**: 30s timeout → IDLE, no harm done
- **"Hey Beemo, never mind"**: Detect dismiss phrases → return to IDLE without processing

## Component 10: User Preferences & Configuration

### Responsibilities
- Store user preferences for personalized responses
- Allow user to register/unregister folders for RAG
- Persist configuration across restarts
- Manage model paths and preferences

### User Preferences
| Preference | Type | Default | Used For |
|------------|------|---------|----------|
| `name` | string | none | Personalized greetings |
| `location` | string | none | Weather, local info, search region |
| `timezone` | string | system timezone | Time-related queries |
| `temp_units` | `fahrenheit` \| `celsius` | fahrenheit | Weather responses |
| `distance_units` | `miles` \| `km` | miles | Distance responses |
| `response_verbosity` | `brief` \| `normal` \| `detailed` | normal | System prompt adjustment |
| `conversation_window_seconds` | int | 30 | How long conversation mode lasts |
| `custom_vocabulary` | list[string] | [] | Names, technical terms for STT hints |

### Config File: `~/.bmo-voice/config.yaml`
```yaml
user:
  name: ""                      # User's preferred name
  location: ""                  # City/region for weather and local queries
  timezone: "auto"              # auto-detect from system, or explicit
  temp_units: "fahrenheit"      # fahrenheit | celsius
  distance_units: "miles"       # miles | km
  response_verbosity: "normal"  # brief | normal | detailed
  custom_vocabulary: []         # names, technical terms

conversation:
  window_seconds: 30            # follow-up listening window after response
  ttl_seconds: 300              # 5 minutes conversation history
  min_speech_duration_ms: 500   # minimum speech to trigger processing in conversation mode

folders:
  - path: /Users/me/Documents/notes
    name: "My Notes"
  - path: /Users/me/projects/myapp
    name: "MyApp Code"

wake_word:
  model_path: ~/.bmo-voice/models/hey_beemo.onnx
  threshold: 0.5

stt:
  model: whisper-small.en      # ◆ reviewable (or whisper-turbo)
  backend: mlx

llm:
  model: qwen3:8b              # ◆ reviewable
  host: http://localhost:11434  # Ollama default

tts:
  model: kokoro-82m             # ◆ reviewable
  voice: default-en

rag:
  embedding_model: bge-small-en-v1.5  # ◆ reviewable
  chunk_size: 512
  chunk_overlap: 50
  top_k: 5

search:
  primary: brave                # ◆ reviewable
  fallback: searxng
  brave_api_key: ${BRAVE_SEARCH_API_KEY}
  searxng_url: http://localhost:8888
  timeout_seconds: 3
```

### Folder Management
- CLI command or simple web UI to add/remove folders
- View indexing status (files processed, chunks created)
- Force re-index button
- Show which folders are currently active
