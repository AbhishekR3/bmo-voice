# Wake Word Detection — `server/wake/`

## Purpose & Responsibilities

- Continuously listen for **"Hey Beemo"** while in IDLE state
- On detection, emit event to session manager to activate the voice pipeline
- Maintain low CPU usage (~2%, ~50MB RAM) during always-on listening
- Reject false positives (TV, music, similar-sounding phrases)

## Architecture

```
                          ┌─────────────────────────┐
                          │   Always-On Loop         │
                          │                          │
Raw PCM Audio ──────────→ │  [OpenWakeWord Engine]   │
(from WebSocket handler)  │   80ms frames            │
                          │   ONNX runtime           │
                          │                          │
                          └───────────┬──────────────┘
                                      │
                              confidence > threshold?
                                      │
                              ┌───────▼────────┐
                              │ Detection Event │
                              │ {confidence,    │
                              │  timestamp}     │
                              └───────┬─────────┘
                                      │
                                      ▼
                              Session Manager
                              (→ LISTENING state)
```

### State-Dependent Behavior

| Session State | Wake Word Detector |
|---|---|
| IDLE | **Active** — only listener running |
| LISTENING | Active — but detections ignored (already activated) |
| PROCESSING | Active — but detections ignored |
| SPEAKING | **Paused** — mic is muted, no audio flows |
| CONVERSATION | Active — "Hey Beemo" resets conversation |

## Technology Choice & Tradeoffs

| Considered | Open Source | Custom Wake Word | Why not chosen |
|---|---|---|---|
| **OpenWakeWord** | Yes (Apache 2.0) | Yes, trainable | **CHOSEN** |
| Porcupine (Picovoice) | No (proprietary) | Yes | Proprietary, free tier limited |
| Mycroft Precise | Yes | Yes | Less actively maintained |

**Why OpenWakeWord:**
- Apache 2.0 license, no API keys, no telemetry
- Custom wake word training from synthetic speech (no need to record thousands of samples)
- Lightweight: ONNX runtime, ~2% CPU, ~50MB RAM
- Python native — easy integration with the rest of the server

**Why not just use Whisper continuously?**
Running Whisper on all audio to check for "Hey Beemo" would consume ~80% CPU constantly. OpenWakeWord is a tiny binary classifier that answers "is this 'Hey Beemo'?" at ~2% CPU. Same approach as Alexa/Google Home/Siri — dedicated lightweight model for wake word, separate from full speech recognition.

**Fallback:** If OpenWakeWord breaks, fall back to keyword spotting via whisper.cpp (continuous short transcripts checking for "hey beemo"). Higher CPU but no additional dependency.

## Custom "Hey Beemo" Model Training

- **Speaker-independent**: Detects the phrase from any speaker, not tied to a specific voice
- **100% synthetic training data**: Generated by TTS models with diverse voices and accents
- Generate ~1000+ synthetic "Hey Beemo" clips using multiple TTS voices
- Generate adversarial negative samples: "hey baby", "hey maybe", "hey people", "hey primo", etc.
- Add room impulse responses and background noise augmentation for robustness
- Train small classifier on OpenWakeWord's frozen feature extractor
- Training takes < 1 hour on CPU, produces lightweight `.onnx` model
- Store at `~/.bmo-voice/models/hey_beemo.onnx`

## False Positive Mitigation

Two strategies combined (recommended):

### Strategy 1: STT Verification
After wake word fires, run a quick STT pass on the audio buffer that triggered it. If the transcript doesn't contain "hey beemo" (or close variant like "hey bmo", "hey beamo"), discard and return to IDLE. Adds ~100-200ms with tiny.en but eliminates most false positives.

### Strategy 2: Confidence Threshold Gating
OpenWakeWord returns a confidence score (0.0–1.0). Start with threshold at 0.5, tune upward to 0.7–0.8 based on empirical false positive rates. Higher threshold = fewer false positives but may reduce detection range.

### Debounce
Ignore duplicate wake word detections within 500ms window.

## Interface with Other Components

| Direction | Component | Data |
|---|---|---|
| **Input from** | WebSocket handler | Raw PCM audio stream (16kHz, 16-bit, mono) |
| **Output to** | Session Manager | Detection event: `{confidence: float, timestamp: int}` |
| **Controlled by** | Session Manager | Pause/resume commands on state changes |
| **Shares with** | Audio Pipeline | Same raw audio stream feeds both wake word and audio pipeline |

## Configuration

```yaml
wake_word:
  model_path: ~/.bmo-voice/models/hey_beemo.onnx
  threshold: 0.5          # confidence threshold (tune 0.5–0.8)
  frame_ms: 80            # OpenWakeWord default frame size
  debounce_ms: 500        # ignore duplicates within this window
  stt_verification: true  # verify via STT after detection
```

## Health Checks

All checks run at startup. **If any check fails, BMO refuses to start.**

1. **Model exists**: Verify `.onnx` file exists at configured path. If missing, error with instructions on how to train/download.
2. **Model loads**: Load into ONNX runtime without error.
3. **Silence test**: Run inference on 1 second of silent audio → confidence must be < 0.1. If not, model may be corrupt.

## Edge Cases

| Scenario | Handling |
|---|---|
| TV/music playing | Threshold tuning + adversarial training + STT verification |
| Multiple rapid detections | 500ms debounce prevents duplicate triggers |
| During TTS playback | Detector paused entirely (mic muted) |
| Model file missing | Startup fails with clear error and instructions |
| Model file corrupt | Silence test catches it at startup |
| "Hey Beemo" from TV | STT verification checks if real speech follows |

## macOS Considerations

- **Microphone permission**: User must grant mic access in System Settings > Privacy & Security > Microphone. If denied, startup fails with clear error message.
- **Audio input device**: Uses system default input device. Configurable if user has multiple mics.
- **Background operation**: When running as daemon via `launchd`, mic permission must be granted to the terminal/app running BMO.
- **Energy**: ~2% CPU is negligible for always-on operation. No thermal concerns.

## Relation to Other Components

- **Upstream**: Receives raw audio from the WebSocket handler (same stream that feeds the audio pipeline)
- **Downstream**: Detection event triggers session manager to transition from IDLE → LISTENING
- **Parallel with**: Audio pipeline — both consume the same audio stream, but wake word runs on raw unprocessed audio while the audio pipeline applies noise reduction
- **Depends on**: Nothing (self-contained with its ONNX model)
- **Depended on by**: Session manager (can't activate without wake word), STT (for verification pass)
